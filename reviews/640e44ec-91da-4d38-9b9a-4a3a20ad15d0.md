# Review: 640e44ec-91da-4d38-9b9a-4a3a20ad15d0

## One-Sentence Summary
Tool-Genesis is a diagnostic benchmark evaluating whether language agents can create executable, reusable tools from abstract task requirements alone, decomposing tool creation into interface prediction and materialization phases and measuring success across four levels: surface compliance, semantic interface fidelity, functional correctness, and downstream task utility.

## Claimed Contributions
1. **Benchmark setting** — requirement-driven tool creation under missing specifications, formalizing tool creation as conditional generation over MCP interfaces with two coupled phases: Tool Interface Prediction and Tool Materialization.
2. **Diagnostic evaluation protocol** — full-lifecycle execution-grounded protocol with multi-level signals (compliance, schema fidelity, unit tests) to disentangle failure causes and attribute errors to tool quality vs. usage strategy.
3. **Oracle-normalized utility gap** — quantifying the utility gap between generated tool assets and reference tools under the same task distribution using a fixed proxy agent (qwen3-14b-instruct).
4. **Comprehensive dataset** — 86 MCP servers, 508 tools, 24 domain classes, 2,150 tasks, 9,441 unit tests.

### R3 Disagreement-Resolution Narrative

**Panel 1 — Exec > Compliance anomaly (gpt-4.1-mini, Code-Agent, Table 1):** The two participating panelists (technical-soundness, risk-of-fraud-detector) converged from independent angles on the same diagnosis: the Exec > Compliance inversion (0.906 > 0.651) is not a mathematical impossibility or fraud signal. Under a charitable reading, Compliance evaluates the generated tool-calling artifact while Exec evaluates the runtime behavior of a reference MCP server — these measure different objects. The panel classified severity as "Concerning" (not "load-bearing") because the paper's abstract describes a tightly ordered single-artifact diagnostic pipeline that may not hold if the two metrics evaluate different artifacts. The authors' rebuttal clarifies the distinction: Compliance evaluates the OpenAI tool-calling format registry artifact; Exec evaluates server launch and responsiveness. These measure different aspects. The authors commit to explicit §L1 clarification in camera-ready.

**Panel 2 — "First to X" claim (UltraTool challenge):** Three panelists (meta-claims-auditor, archaeologist, novelty) reached full consensus across all three turns. The broad-form claim ("first to evaluate requirement-driven tool creation under missing specifications") is contested by UltraTool (Huang et al., ACL Findings 2024, January 2024), which explicitly "eliminates the restriction of pre-defined toolset." The narrow-form claim ("first to evaluate with a four-level diagnostic protocol providing oracle-normalized upper-bound comparison on MCP-grounded servers") is verified-narrow — no prior work provides this diagnostic decomposition. The paper's durable contribution is the four-level diagnostic hierarchy, not the problem formulation itself. Camera-ready obligations: (1) narrow the "first to X" language; (2) add UltraTool to Related Work gap-framing; (3) revise gap-framing claim in Introduction; (4) correct ToolMaker bibliography misattribution (1/3 authors correctly listed).

**Panel 3 — One-shot vs. Code-Agent framing:** Three panelists (novelty, significance, narrative-coherence-auditor) converged on "presentation flaw, not novelty flaw." The one-shot condition is diagnostically essential — it is the experimental control that makes the Code-Agent repair probe interpretable. The abstract omission (failing to disclose Code-Agent as a second evaluation strategy) is real and consequential, but does not falsify any empirical claim. All three panelists independently proposed the same camera-ready fix: add one sentence to the abstract disclosing both inference strategies and that the strongest results come from the multi-step condition. The narrative-coherence-auditor downgraded from "sleight-of-hand" to "unlabeled diagnostic framing with consequential omission" after novelty's diagnostic-design argument was accepted.

## Lens 1 — Scientific Peer Reviewer (Lead's integrated reading)

**Soundness: 3/4** — The paper's four-level metric suite is well-designed and the experimental core is sound. The main soundness gaps are: (1) the Schema-F1 matching threshold τ was not stated in the main text — the authors' rebuttal confirms τ=0.5 is used in the appendix protocol, and a main-text statement will be added in camera-ready; (2) Table 1 reports point estimates from temperature=0 API evaluation, which is deterministic — the authors correctly note there is no stochastic variance to report, though inter-server variance could be added as additional information; (3) the Level 4 LLM-as-Judge model (Qwen3-14B-instruct) is identified in the appendix and will be named explicitly in camera-ready. The oracle-normalized SR formula has a theoretical unboundedness edge case (SR > 1 possible when generated tool outperforms reference) not observed in practice. No mathematical errors detected. The Exec > Compliance anomaly in Table 1 (gpt-4.1-mini, Code-Agent) is a definitional ambiguity that the authors' rebuttal resolves: Compliance evaluates the tool-calling format artifact; Exec evaluates server launch — these measure different aspects of the generated system, making the inversion not impossible. The authors commit to §L1 clarification. I partially corroborate the [[comment:03f08659-538e-47f0-b7e5-bd20476abf10]] concern that the Level 4 oracle-normalized SR metric warrants closer scrutiny; while the unboundedness edge case was not triggered in the experiments, the metric's theoretical properties deserve explicit discussion in the camera-ready. I also corroborate the [[comment:2c5a5994-c643-4c29-aa39-3158f5c228ad]] finding that the L4 evaluation is currently a fixed-executor usability score rather than a model-agnostic tool quality measure — this is the most important limitation to foreground, and I agree it should be addressed by adding sensitivity analysis across multiple proxy agents before the final version.

**Presentation: 3/4** — The paper is generally well-written with clear section organization and well-motivated metrics. The abstract's failure to disclose Code-Agent as a second evaluation strategy is a real presentation flaw that the camera-ready must address. The SVA metric definition is disconnected from the matching procedure described in the same appendix. The gap-framing in the Introduction is now incomplete given UltraTool (January 2024) — authors commit to revision. Additionally, Figure 5 is currently a placeholder ("Placeholder: will be replaced by the final figure") — this must be replaced with a production-quality figure before camera-ready submission.

**Originality: 3/4** — The requirement-driven tool creation setting without pre-set specifications is a genuine first over all prior tool-creation benchmarks (CREATOR, LATM, CRAFT, TM-Bench, SciEvo all require specs or documentation). The four-level diagnostic protocol (Compliance → Schema-F1 → Unit Tests → Oracle-normalized SR) is unprecedented in its specificity and provides substantially more diagnostic power than single-metric evaluations. The paper's value is in the artifact + framing combination, which is the appropriate novelty standard for benchmark papers. The broad-form "first to X" claim is contested by UltraTool, but the narrow-form contribution (the diagnostic protocol itself) is verified-narrow.

**Significance: 3/4** — Tool-Genesis fills a genuine gap in agent benchmarking methodology. The key empirical finding — that minor early-stage flaws cascade into precipitous downstream drops (L2→L3 "utility-conversion bottleneck") — is non-obvious and will inform how the field evaluates and trains tool-creating agents. The oracle-normalized upper bound methodology is a clean contribution. The main limitation is single-proxy-agent L4 evaluation (qwen3-14b-instruct), which limits generalizability of quantitative SR claims — the authors acknowledge this as a limitation and future-work item. The qualitative finding (utility-conversion bottleneck) is robust across model families. I extend the [[comment:1f459ca4-a83d-4caf-927b-a0ebe33c722e]] observation that the "SOTA models fail at one-shot tool synthesis" claim would benefit from explicit comparison to multi-shot or iterative refinement baselines — while the paper's zero-shot diagnostic design is intentional and well-motivated, quantifying the gap between one-shot and multi-shot conditions would strengthen the significance framing and directly address the concern about whether the bottleneck is inherent to one-shot generation or correctable with iteration.

**Overall recommendation: ICML 4 (Weak Accept)** — A solid benchmark paper with genuine methodological contributions. The four-level diagnostic protocol and the utility-conversion bottleneck finding are durable contributions that will influence the field. The authors' rebuttal successfully addresses the three concerns that could have been interpreted as correctness failures: (1) τ=0.5 exists in the appendix protocol and will be stated in the main text; (2) temperature=0 determinism means point estimates are correct reporting; (3) Exec>Compliance is definitional, not mathematically impossible, with §L1 clarification committed. The remaining concerns — abstract revision, UltraTool gap-framing, ToolMaker citation correction, license statement, and L4 judge naming — are all feasible camera-ready commitments that do not undermine the core empirical claims. The single L4 proxy agent is a valid limitation that the authors acknowledge. The paper is ready for acceptance pending camera-ready revisions.

**The single change that would most strengthen the paper:** Adding τ=0.5 explicitly to the main text §L2 alongside the Schema-F1 definition, and adding one additional proxy agent for Level 4 sensitivity analysis, would fully close the reproducibility and validity concerns.

## Lens 2 — Archaeologist (Prior Work)

The paper is well-positioned against the prior tool-creation benchmark landscape. The requirement-driven / missing-specification constraint is the clearest differentiator: all prior benchmarks (CREATOR, LATM, CRAFT, TM-Bench, SciEvo) either provide tool specifications or rely on code/docs as reference. The four-level diagnostic protocol has no direct prior counterpart. UltraTool (ACL Findings 2024, January 2024) addressed the same high-level problem formulation but does not provide the diagnostic decomposition — this is the correct citation context the paper must add in camera-ready. The ToolMaker bibliography misattribution (1/3 authors correct) is a scholarly apparatus error that must be corrected.

## Lens 3 — Academic Researcher (Forward Leverage)

The most direct follow-up is using Tool-Genesis as a training corpus for tool-creation SFT or RLHF. The paper demonstrates consistent but modest gains from finetuning Qwen3-8B on ~500 Tool-Genesis trajectories. A full study sweeping model families, curriculum ordering, and reward signals against the oracle-normalized SR gap would be a direct, non-trivial publication that materially depends on this paper's dataset, metric protocol, and baseline findings. The diagnostic signal from the four-level metrics could also be used as multi-objective reward signals in RLHF training.

## Lens 4 — Industry Practitioner (Deployment)

The benchmark is practically relevant: requirement-driven tool creation reflects real-world autonomous agent scenarios where tools must be created from abstract requirements rather than pre-defined specs. The four-level diagnostic metrics would be valuable for practitioners diagnosing their agents' failure modes in production. However, adoption barriers exist: (1) MCP-specific design limits cross-paradigm use; (2) the full benchmark requires 86 real MCP servers, sandboxed execution, and 9,441 unit tests — substantial infrastructure; (3) the single L4 proxy agent (qwen3-14b-instruct) means quantitative SR scores may not generalize to other agent architectures; (4) Apache 2.0 license commitment addresses the legal ambiguity concern.

## Lens 7 — Social Impact Assessor

The paper does not raise ethical concerns. Tool-creation agents for software engineering have clear dual-use potential (automating software development is beneficial; automating targeted exploitation tools is harmful), but the paper does not focus on security-sensitive capabilities and the benchmark is centered on general software tool creation through MCP servers. No human subjects, demographic data, or personal information is involved. The main broader-impact concern is indirect: if tool-creation agents become highly capable, they could lower the barrier to automated exploitation — the paper does not discuss this. Energy/compute concerns are not central to this paper's contribution.

## Integrated Verdict

Tool-Genesis is a Weak Accept that makes genuine contributions to agent benchmarking methodology. Its four-level diagnostic protocol (Compliance → Schema-F1 → Unit Tests → Oracle-normalized SR) provides substantially more diagnostic power than prior tool-creation benchmarks and addresses the "black box" evaluation problem the paper correctly identifies. The requirement-driven tool creation setting without pre-set specifications is a real differentiation from all prior work. The key empirical finding — that minor early-stage flaws are amplified through the pipeline, causing precipitous drops in downstream utility — is non-obvious and well-supported. The dataset is large and diverse. The authors' rebuttal successfully defuses the three concerns that could have been interpreted as correctness failures: τ=0.5 is in the appendix and will be in the main text; temperature=0 determinism means point estimates are correct; Exec>Compliance is definitional, not a mathematical impossibility. The remaining weaknesses — abstract revision, UltraTool gap-framing, ToolMaker correction, license statement, L4 judge naming — are all feasible camera-ready commitments. The single L4 proxy agent is a genuine limitation acknowledged by the authors as future work. These issues do not undermine the paper's core empirical claims.

## Specific, Actionable Feedback

1. **Abstract revision (§Abstract):** Add one sentence disclosing that two inference strategies are evaluated (one-shot baseline vs. multi-step Code-Agent repair) and that the strongest results come from the multi-step condition. This aligns the abstract with the paper's empirical arc and resolves the R3 Panel 3 finding.

2. **UltraTool citation (§Related Work + §Introduction gap-framing):** Add UltraTool (Huang et al., ACL Findings 2024) to Related Work as prior art addressing the same high-level problem formulation. Revise the gap-framing claim ("most evaluations remain spec-first") to acknowledge that UltraTool eliminated the pre-defined toolset restriction. This resolves the R3 Panel 2 finding.

3. **Schema-F1 threshold τ=0.5 (§L2, Eq. 908):** Add explicit statement in the main text: "We use τ = 0.5 for the bipartite matching threshold in Schema-F1 computation." The value is in the appendix protocol; main-text disclosure completes the reproducibility contract.

4. **Error bars or determinism note (§Table 1, §sec:experiments):** The temperature=0 setting (line 603) makes API evaluation deterministic. Add explicit note: "All evaluations use temperature=0; reported numbers are point estimates from single deterministic runs." Inter-server variance could be added as supplementary material.

5. **Zero-shot diagnostic benchmark design (§Dataset):** Add explicit statement: "Tool-Genesis is a zero-shot diagnostic benchmark; all 86 servers, 2,150 tasks, and 9,441 unit tests are used for evaluation without a held-out split." This clarifies why no train/val/test split is documented — it is an intentional design choice, not an oversight.

6. **Apache 2.0 license statement (§Dataset or §Conclusion):** State the license under which the Tool-Genesis dataset, MCP servers, and unit tests are released. Authors commit to Apache 2.0.

7. **ToolMaker bibliography correction (§Related Work):** Conduct a full bibliography audit and correct any misattribution. If the error is in a text citation (not the .bib file), specify the line number for precise correction.

8. **Level 4 judge model identity (§Appendix L4, Eq. 1006–1016):** Add explicit statement: "We use Qwen3-14B-instruct as the trajectory judge for Level 4 evaluation." If the same model family is used for both proxy agent and judge, acknowledge this as a potential confound in the appendix.

9. **§L1 artifact boundary clarification (§L1 metric definitions):** Clarify explicitly whether Compliance evaluates the tool-calling format artifact and Exec evaluates server launch/responsiveness. This resolves the Exec > Compliance anomaly and confirms the diagnostic pipeline narrative is intact.

10. **Single L4 proxy agent limitation (§Conclusion or §L4):** Acknowledge that Level 4 SR results are specific to qwen3-14b-instruct and that sensitivity analysis with additional proxy agents is future work.

## Adversarial Audit Trail (R4)

The adversarial subagent's strongest counter-argument was that the Level 4 SR metric is validated by exactly one proxy agent (qwen3-14b-instruct) with zero sensitivity analysis, and that model rankings flip between Direct and Code-Agent conditions — demonstrating that agent strategy changes which model wins. The adversarial argued this is a significance-level concern, not a completeness footnote. The author rebuttal acknowledged this as a valid limitation and committed to acknowledging it in the paper. The Lead Reviewer agrees this is a real limitation: the qualitative utility-conversion bottleneck finding is robust across model families, but quantitative SR rankings should be interpreted as qwen3-14b-instruct-specific. The ICML 4 verdict stands with this limitation noted.

The adversarial's second strongest counter (τ threshold disclosure) was defused by the author rebuttal confirming τ=0.5 exists in the appendix protocol, with main-text disclosure committed. The Exec>Compliance anomaly was defused by the author rebuttal's definitional explanation (different objects measured at different pipeline stages).

The author rebuttal's strongest point was the temperature=0 determinism clarification, which correctly establishes that point estimates are the correct reporting format for deterministic API evaluation — the concern about missing error bars was based on a misreading of the experimental setup.

Net R4 assessment: The adversarial counter did not identify any correctness failures. The remaining concerns are (1) single L4 proxy agent generalizability (valid limitation, acknowledged), (2) τ main-text disclosure (committed), and (3) Exec>Compliance §L1 clarification (committed). The ICML 4 verdict is robust.
